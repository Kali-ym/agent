# Transformer 架构详解

> Transformer 不是“大模型里的一个模块”，而是现代 LLM 的基础计算骨架。它用注意力替代循环，把“当前 token 应该看谁、看多少、怎么汇总”变成一套可并行、可扩展的机制。

## 概述

理解 Transformer，重点不是死记公式，而是回答四个问题：

1. **它解决了什么问题**：为什么 RNN/LSTM 不够。
2. **它的最小结构是什么**：输入是怎么变成下一层表示的。
3. **它为什么能扩到大模型**：为什么适合预训练和大规模并行。
4. **它和 Agent 有什么关系**：为什么上下文、工具调用、长文本、KV Cache 都绕不开它。

结合上一节 [[02-agent-history|Agent 技术演进史]]，Transformer 是从“深度学习模型”走向“通用语言底座”的关键一步。没有 Transformer，就不会有今天的 LLM、Coding Agent、Web Agent 和上下文工程。

## 1. 先看它替代了谁

Transformer 出现之前，序列建模主力是 RNN、LSTM、GRU。

| 模型          | 优点             | 问题              |
| :---------- | :------------- | :-------------- |
| RNN         | 顺序建模天然直接       | 必须串行，长依赖弱，梯度难训练 |
| LSTM / GRU  | 缓解长依赖和梯度问题     | 仍然串行，吞吐受限       |
| CNN 序列模型    | 并行性较好          | 长距离依赖需要堆很多层     |
| Transformer | 全局注意力、强并行、扩展性好 | 注意力成本随序列长度平方增长  |

一句话理解差异：

- RNN 是“一个词接一个词往后传”。
- Transformer 是“每个词都可以直接看整段上下文里和自己有关的部分”。

这件事对大规模训练特别重要，因为 GPU 更适合大矩阵并行，不适合长链条串行等待。

## 2. Transformer 的核心直觉

Transformer 的核心不是“记忆前一个状态”，而是“为当前位置动态汇总最相关的信息”。

可以把它理解成一个信息路由器：

```text
输入序列
  -> 为每个位置生成表示
  -> 计算它应该关注哪些位置
  -> 按相关性加权汇总信息
  -> 经过非线性变换继续加工
  -> 输出更高级的表示
```

所以 Attention 本质上是在回答：

- 我现在在处理哪个位置？
- 我需要看上下文里的谁？
- 每个来源该占多大权重？
- 看完之后，我这一位的新表示应该是什么？

这就是现代 LLM 的最小思维骨架。

## 3. 一个 Transformer Block 长什么样

从工程视角看，一个标准 Transformer block 可以粗略拆成：

```text
输入表示
  -> 注意力层（让当前位置和上下文交互）
  -> 残差连接
  -> 归一化
  -> 前馈网络（逐位置做非线性变换）
  -> 残差连接
  -> 归一化
  -> 输出表示
```

更简化的人话版：

| 模块 | 作用 | 你可以把它理解成 |
|:---|:---|:---|
| Token Embedding | 把离散 token 变成向量 | 给每个词一个机器能算的坐标 |
| Position Encoding | 告诉模型顺序 | 给“第几个位置”打标签 |
| Self-Attention | 让不同位置互相看 | 找上下文里和我最相关的信息 |
| Feed Forward Network | 做局部非线性加工 | 把聚合后的信息再提炼一遍 |
| Residual | 保留原始通路 | 防止信息层层传递时丢失 |
| Normalization | 稳定训练 | 控制数值分布，方便深层堆叠 |

一层 block 只完成一次“看别人 + 加工自己”。把很多层堆起来，模型就能从浅层局部模式一路抽象到深层语义、逻辑和任务策略。

## 4. Self-Attention 到底在做什么

Self-Attention 是 Transformer 的核心。

每个位置都会生成三种视角：

| 视角    | 作用          | 类比   |
| :---- | :---------- | :--- |
| Query | 我现在想找什么信息   | 检索请求 |
| Key   | 我这里提供的是什么信息 | 检索索引 |
| Value | 真正要被拿走的信息内容 | 检索结果 |

所以注意力过程可以理解成：

```text
当前位置发出 Query
  -> 和所有位置的 Key 做匹配
  -> 得到一组相关性分数
  -> 分数归一化成权重
  -> 对所有位置的 Value 做加权汇总
  -> 得到当前位置的新表示
```

这比 RNN 更强的地方在于：

- 不需要一跳一跳传递信息。
- 可以直接建立长距离联系。
- 每个位置都能自适应选择信息源。

比如一句话里代词消解、跨句指代、代码里的变量引用、HTML 标签配对，本质上都需要“当前位置去看远处的相关位置”。

## 5. 为什么要除以 \(\sqrt{d_k}\)

注意力分数来自 Query 和 Key 的相似度。如果维度很大，点积值容易变得很大，softmax 就会变得特别尖锐，训练梯度会不健康。

所以缩放的作用很朴素：

- **不缩放**：分数可能过大，注意力接近 one-hot，训练不稳定。
- **缩放后**：分布更平滑，梯度更健康，优化更稳定。

这一点在面试里常被问，但工程上你只要记住：**这是为了让注意力分数处在适合 softmax 的数值范围里。**

## 6. Multi-Head Attention 为什么不是一个大头

如果只用一个注意力头，模型每次只能形成一种“关注模式”。

多头注意力的意义是：让模型从多个子空间、多个关系视角同时看上下文。

你可以这样理解：

- 一个头看语法依赖。
- 一个头看实体指代。
- 一个头看局部搭配。
- 一个头看长距离逻辑关系。
- 一个头在代码里看变量引用。
- 一个头在表格里看列之间的对齐关系。

它不是简单“把模型切几份”，而是给系统多种并行的相关性视角。

| 方案 | 特点 | 问题 |
|:---|:---|:---|
| 单头 | 结构简单 | 表达视角单一 |
| 多头 | 能捕获不同关系子空间 | 计算和缓存开销更高 |

后面很多推理优化，本质上都在回答：怎样尽量保留多头表达力，同时减少显存和延迟。

## 7. 位置编码为什么必需

Attention 自己并不知道顺序。

如果没有位置编码，模型更像是在看一个“集合”，知道有哪些 token，但不知道它们前后关系。语言、代码、表格、网页 DOM 都强依赖顺序，所以必须显式注入位置信息。

### 常见位置方案

| 方案 | 核心思路 | 优势 | 局限 |
|:---|:---|:---|:---|
| Sin/Cos 绝对位置编码 | 用固定周期函数编码位置 | 无需学习，外推性较好 | 表达相对僵硬 |
| Learned Position Embedding | 每个位置一个可学习向量 | 灵活 | 超出训练长度时外推差 |
| RoPE | 用旋转方式把位置注入 Q/K | 相对位置感知强，长文本友好 | 实现和调参更复杂 |
| ALiBi | 用线性偏置建模距离衰减 | 简洁、长文本友好 | 表达能力有取舍 |

原始论文用的是 Sin/Cos，但现代主流 LLM 更多采用 **RoPE**。

原因也很现实：

- 大模型越来越依赖长上下文。
- 代码、Agent 轨迹、工具结果、网页内容都很长。
- 相对位置比绝对位置更符合“当前 token 看前后多远信息”的需求。

这也是为什么后面很多模型系列都会重点讨论 RoPE、长上下文扩展和位置外推。

## 8. Feed Forward Network 在干什么

很多人只盯 Attention，但 Transformer 里另一个大块是前馈网络，也就是 MLP/FFN。

它的作用不是“再做一次注意力”，而是：

- 对每个位置的表示单独做非线性加工。
- 把注意力聚合来的信息重新映射到更有表达力的空间。
- 提供大量参数容量，承载知识和模式。

从参数分布上看，现代大模型里 **MLP 往往占很大一部分参数量**。所以：

- Attention 决定“信息从哪来”。
- FFN 决定“信息如何变形和存储”。

一句话记忆：

```text
Attention 负责路由信息
FFN 负责加工信息
```

## 9. Residual 和 Normalization 为什么不可少

Transformer 能堆很多层，不只是因为 Attention 聪明，更因为训练结构设计得稳。

### Residual

残差连接相当于给每层留一条直通通道：

- 原始信息不会轻易被下一层完全覆盖。
- 深层网络更容易训练。
- 梯度更容易向前传播。

### LayerNorm / RMSNorm

归一化的作用是控制表示尺度，让不同层的数值分布更稳定。

在原始 Transformer 里常见的是 **LayerNorm**，而在很多现代 LLM 里会看到 **RMSNorm**。

| 方案 | 特点 | 常见场景 |
|:---|:---|:---|
| LayerNorm | 经典稳定 | 原始 Transformer、BERT、很多早期模型 |
| RMSNorm | 更轻，去掉均值中心化 | LLaMA、Qwen、很多新模型 |

这类变化不是“推翻 Transformer”，而是在保留主骨架的前提下做工程优化。

## 10. Encoder、Decoder、Decoder-only 的区别

原始 Transformer 论文是 **Encoder-Decoder** 结构，适合机器翻译：

```text
源序列
  -> Encoder 编码
  -> Decoder 一边看已生成内容，一边看编码结果
  -> 逐步生成目标序列
```

后来不同任务形成了三类主流结构：

| 架构 | 代表任务 | 特点 |
|:---|:---|:---|
| Encoder-only | 理解类任务 | 擅长分类、检索、表示学习 |
| Encoder-Decoder | Seq2Seq 任务 | 输入输出分工清楚，适合翻译、摘要 |
| Decoder-only | 自回归生成 | 最适合通用 LLM 和聊天/代码生成 |

### 为什么今天 LLM 多数是 Decoder-only

因为它更适合统一很多任务：

- 对话生成
- 补全
- 代码续写
- 工具调用参数生成
- Agent 的下一步动作决策

在 Agent 场景里，模型经常要根据当前上下文预测“下一个 token / 下一个动作描述”，所以 Decoder-only 很自然地成为主流底座。

## 11. Causal Mask：为什么生成模型不能偷看未来

在自回归生成里，模型预测当前位置时，不能直接看到未来 token，否则训练和推理条件不一致。

这时需要 **Causal Mask**：

- 只能看当前及之前的位置。
- 不能看后面的答案。

这件事对聊天、代码补全、工具调用轨迹都很关键。

你可以把它看成一个硬规则：

```text
当前位置
  -> 可以读历史
  -> 不能偷看未来
```

而在编码器里，通常使用双向可见，因为理解任务需要同时看前后文。

## 12. 为什么 Transformer 能成为大模型底座

Transformer 之所以统治 LLM，不只是因为效果好，而是因为它和现代计算资源高度匹配。

### 它的四个底层优势

| 能力 | 价值 |
|:---|:---|
| 并行训练 | 更适合 GPU/TPU 大矩阵运算 |
| 可扩展堆叠 | 更容易加层、加宽、加数据 |
| 通用输入表达 | 文本、代码、图像 patch、音频 token 都能 token 化处理 |
| 统一生成接口 | 训练目标和推理方式能统一成“预测下一个 token” |

这使它特别适合预训练：

```text
海量语料
  -> token 化
  -> Transformer 建模上下文关系
  -> 学到通用表示和生成能力
  -> 再微调 / 对齐 / 指令化
  -> 变成可用的大模型
```

所以你今天看到的 LLaMA、Qwen、DeepSeek，本质上仍然是在 Transformer 主体上演化。

## 13. 现代 LLM 对 Transformer 做了哪些改造

现代模型基本不是原封不动地照抄 2017 论文，而是在几个方向上持续改造。

### 位置与长度

- Sin/Cos -> RoPE
- 更强的长上下文扩展
- 更关注位置外推和长度稳定性

### 归一化与激活

- LayerNorm -> RMSNorm
- ReLU -> GELU / SwiGLU
- 目标是更稳、更省、更强表达

### 注意力效率

- MHA -> GQA / MQA
- FlashAttention 做 IO 优化
- 各类线性 / 稀疏 / 压缩注意力尝试降低长序列成本

### 容量扩展

- Dense -> MoE
- 让参数规模变大，但单次激活成本不必等比增加

### 推理工程

- KV Cache
- Prefill / Decode 分阶段优化
- 分页缓存、并行采样、量化、张量并行

可以把现代 LLM 理解成：

```text
Transformer 主骨架
  + 更好的位置编码
  + 更省的注意力
  + 更稳的归一化与激活
  + 更强的推理系统
  = 今天的大模型
```

## 14. KV Cache 为什么是推理高频词

在 Decoder-only 生成时，每次新生成一个 token，都需要参考之前所有 token 的 Key 和 Value。

如果每一步都把历史重新算一遍，成本会非常高。所以推理时通常会缓存历史 K/V，这就是 **KV Cache**。

它带来的变化：

| 没有 KV Cache | 有 KV Cache |
|:---|:---|
| 每一步重复算历史 | 历史只算一次，后面复用 |
| 延迟高 | 解码更快 |
| 计算浪费大 | 空间换时间 |

但代价也明确：**显存压力会上升**。这就是为什么长上下文模型、Agent 长轨迹、工具多轮调用，经常会遇到 KV Cache 成本问题。

这也直接影响 Agent 工程：

- 上下文越长，预填充越慢。
- 历史轨迹越长，缓存越大。
- 如果不做摘要、裁剪、记忆分层，成本会迅速变高。

所以 Transformer 不是只影响模型内部，它会直接塑造上层的上下文工程策略。

## 15. Transformer 和 Agent 的关系

这门课题放在 Agent 项目里，不是为了单独学模型，而是为了理解上层系统为什么这么设计。

### Agent 中很多工程问题，本质上是 Transformer 约束

| Agent 现象           | Transformer 侧原因                    |
| :----------------- | :--------------------------------- |
| 长上下文成本高            | Attention 和 KV Cache 成本随长度上升       |
| 需要 RAG / Memory 分层 | 不能把所有历史都一直塞进上下文                    |
| 需要结构化输出            | Decoder-only 本质是在按 token 生成，需要格式约束 |
| 需要工具摘要             | 原始工具输出太长，会挤占注意力预算                  |
| 需要中断恢复             | 长轨迹会拖慢推理，状态不能只靠全文上下文保存             |
| 需要评测不同 prompt 结构   | 输入顺序会影响注意力分配和生成行为                  |

## 16. 常见面试追问

### Q1：Transformer 为什么比 RNN 更适合长序列？

标准回答方向：

- 长距离依赖不需要逐步传递。
- 训练可并行。
- 更适合大规模数据和算力扩展。
- 但注意 Attention 成本是 \(O(n^2)\)，长序列不是完全没有代价。

### Q2：多头注意力为什么有效？

标准回答方向：

- 不同头学习不同关系模式。
- 提升表达力，而不是简单复制。
- 代价是计算和缓存开销增加。

### Q3：为什么需要位置编码？

标准回答方向：

- Attention 天然不感知顺序。
- 语言和代码强依赖顺序。
- 所以必须显式注入位置信息。

### Q4：KV Cache 在解决什么问题？

标准回答方向：

- 避免自回归解码时重复计算历史 K/V。
- 本质是空间换时间。
- 对长上下文和高并发推理特别关键。

### Q5：现代 LLM 为什么爱用 RoPE、RMSNorm、SwiGLU、GQA？

标准回答方向：

- 它们不是替代 Transformer，而是在保留主骨架的前提下做更适合长文本、推理效率和训练稳定性的工程优化。

## 17. 学这一篇时最该记住什么

如果你只记五件事，建议记这五个：

1. **Transformer 用 Attention 替代循环，核心优势是全局交互和并行训练。**
2. **一个 block 的核心分工是：Attention 负责路由，FFN 负责加工。**
3. **位置编码是必须的，因为 Attention 自己不懂顺序。**
4. **现代 LLM 仍是 Transformer 主骨架，只是在位置、归一化、注意力效率和推理工程上持续改造。**
5. **Agent 的上下文工程、Memory、RAG、KV Cache 成本，本质上都和 Transformer 的工作方式有关。**
