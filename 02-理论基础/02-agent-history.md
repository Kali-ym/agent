# Agent 技术演进史

> Agent 的演进不是“聊天机器人越来越聪明”这么简单，而是从规则系统、搜索规划、强化学习、深度学习、大语言模型，逐步走向“模型 + 工具 + 状态 + 记忆 + 环境反馈 + 评测”的工程系统。

## 概述

理解 Agent 技术史，重点不是背年份，而是看清三个问题：

1. **决策权从哪里来**：从人工规则，到统计模型，再到 LLM 根据上下文选择动作。
2. **环境反馈怎么进入系统**：从封闭规则库，到交互式环境，再到真实浏览器、代码仓库、数据库和企业工具。
3. **系统可靠性靠什么保证**：从专家规则，到 benchmark，再到 trace、eval、权限、回滚和 human-in-the-loop。

和前一篇 [[01-what-is-agent|什么是 AI Agent？]] 对应，现代 Agent 可以理解为：把 LLM 放进一个有目标、有状态、有工具、有约束、有反馈的执行循环中。

## 一张时间线

| 阶段 | 时间 | 核心形态 | 代表能力 | 主要局限 |
|:---|:---|:---|:---|:---|
| 符号主义与搜索 | 1950s-1970s | 规则、逻辑、搜索树 | 可解释推理、棋类搜索、规划 | 依赖人工建模，迁移能力弱 |
| 专家系统 | 1970s-1980s | 规则库 + 推理机 | 垂直领域决策支持 | 知识维护成本高，难处理开放任务 |
| 经典规划与强化学习 | 1980s-2010s | MDP、POMDP、策略学习 | 在环境中试错、学习策略 | 状态空间难建模，泛化成本高 |
| 深度学习 Agent | 2010s | 深度网络 + RL | 游戏、机器人、感知控制 | 数据和算力要求高，语言交互弱 |
| Transformer 与预训练模型 | 2017-2021 | 大规模语言模型 | 通用语言理解、生成、迁移 | 主要停留在文本生成，行动能力弱 |
| LLM Tool Use | 2022-2023 | LLM + 工具调用 | 搜索、计算、代码、API 调用 | 容易循环、幻觉工具结果、缺少边界 |
| 工程化 Agent | 2024-2026 | 上下文工程 + 记忆 + 评测 + 权限 | Coding Agent、Web Agent、企业 Agent | 可靠性、安全、成本、评测仍是核心难题 |

## 1. 符号主义：Agent 的最早想象

早期 AI 更接近“会推理的规则系统”：人把世界拆成符号、规则和目标，系统通过搜索或逻辑推导找到下一步动作。

典型模式：

```text
目标
  -> 当前状态
  -> 可选动作
  -> 搜索 / 推理
  -> 执行动作
  -> 新状态
```

这已经具备 Agent 的雏形：目标、状态、动作、反馈。但它的问题也很明显：现实世界很难被完整写成规则。一旦环境变化，系统就需要大量人工维护。

这一阶段留下的核心资产不是某个具体系统，而是几个长期有效的概念：

- **State**：当前世界是什么样。
- **Action**：系统可以做什么。
- **Policy**：在某个状态下应该选择什么动作。
- **Search / Planning**：如何从当前状态走到目标状态。

这些概念在今天的 Agent loop、LangGraph 状态图、workflow 编排里仍然存在。

## 2. 专家系统：把知识写进规则库

专家系统把领域专家的经验固化成规则库，再由推理机做判断。

常见结构：

```text
用户输入
  -> 事实抽取
  -> 规则匹配
  -> 推理链
  -> 建议 / 诊断 / 决策
```

它的价值在于：在边界明确的领域里，规则系统稳定、可解释、可审计。

它的失败也很典型：

- 知识获取成本高，专家经验很难完整结构化。
- 规则之间容易冲突，维护复杂。
- 面对开放语言输入和新场景时弹性不足。

现代 Agent 的 guardrails、policy、工具权限分级，本质上仍然继承了专家系统的一部分思想：不是所有决策都交给模型，关键边界要显式写出来。

## 3. 规划与强化学习：从“写规则”到“学策略”

随后 Agent 研究更多转向环境交互：系统不只是按规则推理，而是在环境中选择动作、观察结果、更新策略。

这一阶段常见关键词：

- **MDP / POMDP**：把任务建模成状态、动作、奖励和转移。
- **Planning**：在行动前推演多步路径。
- **Reinforcement Learning**：通过奖励信号学习策略。
- **Exploration vs Exploitation**：在探索新路径和利用已知策略之间权衡。

这条路线在游戏、机器人、控制系统中很强，但迁移到通用任务时会遇到两个问题：

1. 真实任务很难定义稳定的奖励函数。
2. 语言、工具、网页、代码仓库这类环境太开放，状态空间巨大。

所以在 LLM 出现前，Agent 很多时候更像研究系统，而不是普通开发者可以快速落地的应用形态。

## 4. Transformer：通用语言能力成为底座

2017 年 Transformer 出现后，预训练语言模型逐步成为通用接口。它改变了 Agent 的基础能力：系统不再只能处理结构化状态，也能直接理解自然语言、文档、网页、代码和工具说明。

这对 Agent 的影响很大：

| 变化 | 对 Agent 的意义 |
|:---|:---|
| 长文本理解 | 可以把任务说明、历史记录、工具文档放进上下文 |
| 代码能力 | 可以读写代码、修复错误、调用开发工具 |
| 指令跟随 | 用户可以用自然语言给目标和约束 |
| 结构化输出 | 模型可以返回 JSON、工具参数、计划步骤 |
| Few-shot 泛化 | 不必为每个任务单独训练模型 |

但这一阶段的模型主要还是“生成文本”。它能回答“应该怎么做”，但不一定能真的去做。Agent 的下一步突破来自工具调用和行动闭环。

可继续阅读：[[03-transformer|Transformer 架构详解]]。

## 5. LLM + Tool Use：模型开始行动

当 LLM 可以稳定地产生结构化工具调用后，Agent 从“会说”走向“会做”。

典型闭环变成：

```text
用户目标
  -> 构造上下文
  -> 模型选择工具
  -> 工具返回观察结果
  -> 模型更新判断
  -> 继续 / 停止 / 请求人工确认
```

这时出现了很多关键模式：

- **Function Calling**：把外部 API 暴露给模型。
- **ReAct**：让模型在推理和行动之间交替。
- **Plan-and-Execute**：先拆任务，再逐步执行。
- **Reflection**：执行失败后复盘并修正。
- **RAG**：通过检索把外部知识注入上下文。

这一阶段最容易产生误解：以为“给模型很多工具”就是 Agent。实际上工具越多，越需要工具说明、权限分级、错误处理、停止条件和 trace。

可继续阅读：[[04-react-framework|手撕 ReAct 框架]]、[[05-cot-and-planning|Chain-of-Thought 与规划]]。

## 6. 从 AutoGPT 热潮到工程冷静期

2023 年前后，AutoGPT、BabyAGI 等项目让“自主 Agent”快速出圈。它们展示了 LLM 可以自动拆任务、搜索资料、写文件、调用工具。

但很快也暴露出问题：

| 问题 | 表现 |
|:---|:---|
| 目标漂移 | 执行几轮后偏离原始任务 |
| 无限循环 | 重复搜索、重复修改、重复自我反思 |
| 证据不足 | 没有真实观察结果就给结论 |
| 成本失控 | 长任务消耗大量 token 和工具调用 |
| 权限风险 | 删除、提交、支付、发消息等动作缺少确认 |
| 难以评测 | demo 看起来能跑，但稳定成功率不清楚 |

这段历史的价值在于让行业认识到：Agent 不是越自主越好，而是要把自主性限制在可观察、可恢复、可评测的边界内。

这也对应项目里的一个核心判断：**很多任务优先用 workflow，只有当路径无法提前写死时，才引入更自主的 Agent loop。**

## 7. 上下文工程：Agent 开发的核心转向

当前项目在技术栈部分强调：**Agent 开发的本质是上下文工程**。

原因很直接：LLM 的行为高度依赖它看到什么、以什么顺序看到、用什么格式看到，以及哪些信息被排除在外。

现代 Agent 的上下文通常由这些部分组成：

| 上下文来源 | 作用 |
|:---|:---|
| System Prompt | 定义角色、边界、优先级和输出协议 |
| User Prompt | 定义当前任务目标 |
| Short-term Memory | 保存当前任务轨迹和临时状态 |
| Long-term Memory | 保存用户偏好、历史经验和可复用知识 |
| RAG | 注入外部知识、文档和业务数据 |
| Tool Results | 注入真实环境反馈 |
| Structured Output Schema | 约束模型输出格式，降低解析失败 |

所以工程重点从“写一个聪明 prompt”转向：

```text
信息源
  -> 筛选
  -> 排序
  -> 压缩
  -> 格式化
  -> 注入模型
  -> 记录 trace
  -> 用 eval 验证效果
```

可继续阅读：[[../03-技术栈/11-context-engineering-practices|Context Engineering Practices]]、[[../03-技术栈/18-context-engineering-guide|Context Engineering Guide]]。

## 8. Agent Harness：从能力演示到可运行系统

当 Agent 进入真实生产环境后，问题不再是“模型能不能做一次”，而是“系统能不能稳定地反复做”。

这时需要 Agent Harness，也就是承载模型运行的工程外壳。

项目里的七层模型可以简化理解为：

| 层级  | 关注点                              |
| :-- | :------------------------------- |
| 模型层 | provider 抽象、模型切换、降级              |
| 循环层 | agent loop、interrupt、resume、最大步数 |
| 工具层 | tools、skills、MCP、权限和错误描述         |
| 记忆层 | JSONL、SQLite、vector store、状态恢复   |
| 人格层 | system prompt、policy、项目规则        |
| 通道层 | CLI、Web、IM、IDE、API               |
| 自驱层 | heartbeat、cron、后台任务和长期运行         |

这一阶段的关键词是：

- **可恢复**：中断后能从状态继续。
- **可观测**：每一步工具调用和状态变化都能追踪。
- **可控权**：高风险动作需要人工确认。
- **可评测**：有固定案例证明成功率、成本和风险变化。
- **可替换**：模型、工具、检索、存储都能独立演进。

可继续阅读：[[../03-技术栈/27-agent-harness-engineering|Agent Harness Engineering]]、[[../03-技术栈/12-factor-agent-architecture|12 Factor Agent Architecture]]。

## 9. 评测、安全与合规成为主线

Agent 越能行动，越需要评测和安全边界。

一个普通文本生成系统出错，通常只是回答错；一个 Agent 出错，可能会改文件、调用 API、泄露数据、误发消息、产生错误交易。

因此现代 Agent 的成熟度不只看模型能力，还要看：

| 维度 | 要回答的问题 |
|:---|:---|
| 成功率 | 在固定任务集上完成率是多少？ |
| 工具正确率 | 是否选择了正确工具和参数？ |
| 证据可靠性 | 结论是否来自真实观察结果？ |
| 成本与延迟 | 每个任务平均消耗多少？ |
| 权限控制 | 高风险动作是否被拦截或确认？ |
| 恢复能力 | 失败后能否重试、回滚或继续？ |
| 安全性 | 是否能抵抗 prompt injection、数据泄露和越权工具调用？ |

这也是为什么项目里单独放了评测、安全、sandbox、MCP、memory 等主题。Agent 工程不是单点能力，而是一组边界清晰的系统能力。

可继续阅读：[[08-agent-bench|AgentBench 详解]]、[[09-evaluation-metrics|Agent 评估指标]]、[[../04-实践指南/03-agent-security|Agent 安全实践]]、[[../03-技术栈/24-agent-sandbox-guide|Agent Sandbox Guide]]。

## 10. 当前趋势：从“万能 Agent”回到“场景 Agent”

到 2026 年，更务实的方向不是做一个能做所有事的 Agent，而是做边界清楚、工具清楚、评测清楚的场景 Agent。

几个主要方向：

| 方向 | 典型任务 | 核心难点 |
|:---|:---|:---|
| Coding Agent | 读代码、改代码、跑测试、修 CI | 工作区状态、补丁质量、回归验证 |
| Web Agent | 浏览网页、填表、跨页面操作 | DOM 变化、登录态、视觉理解、误点击 |
| Research Agent | 搜索资料、读论文、生成综述 | 证据引用、去重、来源质量 |
| Enterprise Agent | 连接 CRM、知识库、工单、审批 | 权限、审计、数据隔离、流程合规 |
| RAG Agent | 检索、重排、生成、追问 | 数据质量、上下文污染、评测集构造 |
| Multi-Agent | 多角色协作完成复杂任务 | 协议、分工、冲突解决、成本控制 |

项目中的学习路线也更适合沿这个方向推进：先理解 Agent 最小闭环，再掌握 ReAct、上下文工程、记忆、评测、安全，最后做可展示、可复现、可评估的项目。

## 关键认知变化

| 旧认知 | 新认知 |
|:---|:---|
| Agent = 自动拆任务 | Agent = 受控环境里的行动系统 |
| Prompt 写好就行 | 上下文、工具、状态、权限、评测同样重要 |
| 工具越多越强 | 工具越多越需要 schema、权限和错误边界 |
| 自主性越高越先进 | 能稳定完成任务才先进 |
| Demo 能跑就算成功 | 固定评测集上稳定复现才算成功 |
| 多 Agent 更智能 | 多 Agent 只有在职责分工清晰时才值得用 |

## 学习建议

如果你是从零开始，不建议直接追复杂框架。更稳的顺序是：

1. 先读 [[01-what-is-agent|什么是 AI Agent？]]，建立 Agent、Workflow、Chatbot 的边界。
2. 再读 [[04-react-framework|ReAct]]，理解 observe -> act -> observe 的最小闭环。
3. 然后学习 [[../03-技术栈/11-context-engineering-practices|上下文工程]]，理解为什么 Agent 开发核心是信息流控制。
4. 接着补 [[../03-技术栈/15-agent-memory|Agent Memory]]、[[../03-技术栈/28-mcp-protocol|MCP]]、[[../03-技术栈/26-agent-evaluation-harness-guide|Evaluation Harness]]。
5. 最后做 [[../07-项目与示例/projects/README|项目示例]]，用 trace 和 eval 证明系统真的有效。

## 一句话总结

Agent 技术史的主线，是从“人写规则让机器推理”，走向“模型在受控工程系统中调用工具并根据反馈行动”。真正决定 Agent 能否落地的，不只是模型智力，而是上下文、工具、状态、权限、记忆、评测和恢复机制是否被系统化设计。
